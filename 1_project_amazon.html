Amazon Market Analysis — End-to-End Data Pipeline

An end-to-end data analysis project simulating a real-world market research workflow: extracting live product data via the Rainforest API, transforming it with Python, modeling it in PostgreSQL, and visualizing it in an interactive Power BI dashboard.

Project overview

This project analyzes top-performing products across four major Amazon categories — Electronics, Beauty & Personal Care, Home & Kitchen, and Sports & Outdoors — to simulate a real-world market analysis workflow. The goal was to take raw e-commerce data and turn it into structured, decision-ready insights, the same way a data analyst would approach a market research request.

I initially explored a Kaggle dataset for this, but the regional data wasn't representative of the current US market, so I pivoted to live extraction via the Rainforest API instead — treating real-time data collection as the differentiator of the project rather than a shortcut around it.

Objective & data pipeline

The objective was to identify the patterns behind top-performing Amazon products by analyzing sales distribution, pricing strategy, review behavior, discount usage, and category-level performance.

Data source

Data was collected using the Rainforest API. Two datasets were built: search_clean, a market-level dataset with ~932 products scraped across categories and search pages (price, rating, review count, sales approximation, discount info, search position), and products_top, a product-level enrichment dataset with brand, subcategory, seller info, Best Seller Rank, and engagement metrics.

Pipeline

API (Rainforest) → JSON → Parsing → Structured Dataset → EDA → SQL → Power BI

Notebook structure

The pipeline was split into five notebooks, each with a single responsibility:

01_api_search.ipynb

Extracts product data across categories and search result pages via the Rainforest API.

02_api_product.ipynb

Enriches top-performing products with deeper, product-level attributes.

03_parse_search.ipynb

Parses raw JSON responses into a structured, analysis-ready dataset.

04_parse_product.ipynb

Processes the enriched product data into its final structured form.

05_eda.ipynb

Exploratory data analysis — distributions, correlations, and category comparisons using pandas, matplotlib and seaborn.

Process

Information architecture

I started using CSVs files from Kaggle for large databases, but the information was old and not for the US market. API Rainforest gave me the real time info by web scrapping the amazon.com site. With this i use all the tokens to store all the data that i can then i did the below steps.

1

Data extraction

Live product data pulled via Rainforest API across 4 categories and multiple search pages. ~932 products collected at the search level.

2

Parsing & structuring

Raw JSON responses parsed into clean, structured datasets using pandas — separating market-level data from product-level enrichment.

3

Exploratory analysis

EDA notebook covering sales distribution, pricing behavior, ratings, discount strategy, and organic vs. sponsored positioning.

4

SQL modeling

Relational schema built in PostgreSQL. Analytical queries using CASE WHEN, JOINs with aggregated subqueries, and window functions (RANK() OVER PARTITION BY). ~7 views created as a semantic layer for BI.

5

Power BI dashboard

Four-page interactive dashboard (Market Analysis, Category Analysis, Top Products, Product Detail drill-through) consuming the SQL views, with KPI cards and dynamic filtering.

Key design decisions

Choosing live data over Kaggle

A prior approach using a Kaggle CSV was discarded because the regional data wasn't representative of the current US market. Switching to the Rainforest API meant building the extraction logic myself, but it made the analysis reflect real, current market conditions rather than a static historical snapshot.

Window functions for category ranking

To extract the top 10 products per category in a single query, I used RANK() OVER (PARTITION BY category) instead of running separate queries per category — a cleaner and more scalable approach for the kind of comparison a stakeholder would actually ask for.

A semantic layer for Power BI

Rather than connecting Power BI directly to raw tables, I built ~7 SQL views that pre-aggregate and structure the data specifically for BI consumption. This keeps the DAX layer simpler and the dashboard logic decoupled from the underlying schema.

Key insights

The redesign shipped in week 6, exactly on schedule. Metrics were tracked over the following 6 weeks post-launch compared to the same prior period.

  • Sales distribution: most products sell between 50–300 units/month, following a long-tail pattern — a small group reaches 1K+ sales
  • Category performance: Beauty & Personal Care drives ~60% of total sales and needs fewer discounts to generate demand; Sports & Outdoors is the weakest performer
  • Ratings: average rating is consistent across categories (~4.4–4.5), suggesting rating alone isn't a strong differentiator
  • Discounts: ~50% of products overall use discounts, but Beauty relies on them far less (~25% vs. ~75% without)
  • Pricing: Electronics is a clear outlier — median price ~$80, range up to $1000+ — while other categories sit around $15–20 with low variability
  • Visibility: ~80% of top-ranking products appear organically vs. ~20% sponsored

Reflections

The biggest lesson from this project was about data quality over data volume: switching from a static Kaggle dataset to live API data was more work, but it's what made the insights actually trustworthy. A polished dashboard built on unrepresentative data isn't useful to anyone.

If I were to extend this project, I'd automate the extraction pipeline to run on a schedule, expand the dataset size, and explore basic sales forecasting on top of the existing structure.
